# Domain docs

本仓库使用 single-context 领域文档布局。工程技能在理解代码、设计变更、诊断问题或编写测试前，应按本文件读取领域上下文。

## 开始探索前读取

- 根目录 `CONTEXT.md`
- 与当前任务相关的 `docs/adr/` 中的 ADR

如果这些文件不存在，继续执行即可。不要因为缺少领域文档而中断任务，也不要预先建议创建；`domain-modeling` 相关流程会在真正需要沉淀术语或决策时再创建或更新它们。

## 文件结构

```text
/
├── CONTEXT.md
├── docs/adr/
│   └── 0001-ended-events-are-final.md
└── ...
```

## 使用领域词汇

当输出中需要命名领域概念时，例如 issue 标题、重构建议、问题假设或测试名称，应使用 `CONTEXT.md` 中定义的术语。不要漂移到 glossary 明确避免的同义词。

如果需要的概念还没有出现在 `CONTEXT.md` 中，先判断自己是否在发明项目并不使用的语言；如果确实是领域缺口，可以记录给后续 `domain-modeling` 流程处理。

## 标出 ADR 冲突

如果你的建议或实现会违背已有 ADR，必须明确指出冲突，而不是静默覆盖既有决策。
